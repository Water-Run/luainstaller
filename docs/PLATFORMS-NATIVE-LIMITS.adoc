[#platforms-contract]
== Native host

The product removes the need for a separate system `lua` command. It does not
translate ABIs, emulate other operating systems, or cross-compile. Build and
run on the target-class environment.

Align: OS family and loader conventions, CPU architecture, system/compiler
ABI, Lua major/minor ABI, and native modules with their system dependencies.

There is no cross-build mode. `--target-os` only asserts the detected host OS;
mismatch → `UnsupportedPlatformError`.

[#platforms-lua]
== Lua

* official Lua `>= 5.1` and `< 6.0`
* LuaJIT is rejected
* analysis interpreter, tracer, headers, linked library, and bundled C modules:
  one major/minor
* launchers are generated for the active ABI; a native linked-runtime probe
  before publish requires `_VERSION` to match the build interpreter

[#platforms-toolchains]
== Toolchains

[cols="1,2,3",options="header"]
|===
|Host |Runtime strategy |Typical requirements

|Linux
|Shared library copied into `.luai/native`
|C compiler, matching-ABI `pkg-config` and shared liblua

|macOS
|Static `liblua.a` linked into the launcher
|Apple compiler, prefix with `lua.h` and `lib/liblua.a`

|Windows
|ABI-specific DLL beside the executable and under `.luai/native`
|MSVC, Windows SDK, headers, import library, matching DLL
|===

When automatic discovery is insufficient, use `--lua-prefix` or
`LUAI_LUA_PREFIX`. An explicit prefix is authoritative; discovery does not
silently substitute another install. The prefix is for the current host and
ABI, not a cross SDK.

A static-only Lua install without a shared library cannot satisfy the
shared-runtime Linux profile. Custom installs may set `PKG_CONFIG_PATH`.

macOS does not produce universal binaries, code-sign, or notarize. Windows
builds and probes run natively; Wine/MinGW are not part of the release path.
Generated runtime code currently uses narrow Win32 path/process APIs.

[#platforms-native]
== Native Lua C modules

Discovery copies the file selected from `package.cpath`. It does not rebuild
for another OS/arch/ABI, rewrite RPATH, recursively package system libraries,
or audit licenses.

[source,bash]
----
LUA_PATH='/srv/app/share/lua/5.x/?.lua;;' \
LUA_CPATH='/srv/app/lib/lua/5.x/?.so;;' \
luai -t app/main.lua --max-deps 250
----

A successful launcher compile only proves the Lua runtime link, not closure of
every application module's system dependencies.

[#platforms-limits]
== Out of scope

* cross-compilation / one binary for many platforms
* automatic closure of all native system libraries
* bytecode-only packaging, encryption, obfuscation
* loading native modules from single-file without extraction
* macOS universal / signing / notarization
* MSI / PKG / DEB / RPM generation

Single-file is an executable distribution form (extract then start), not an OS
installer.
