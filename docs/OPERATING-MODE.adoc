[[overview]]
== Overview

`luainstaller` turns a Lua entry script into a same-environment executable
bundle. Conceptually, it does four jobs:

[arabic]
. Analyze the entry script.
. Record the dependency and compatibility contract.
. Generate a native launcher.
. Assemble an onedir layout or onefile wrapper.

[source,text]
----
entry.lua
   |
   v
dependency discovery
   |
   v
manifest + embedded Lua payload
   |
   v
native launcher
   |
   v
onedir bundle or onefile wrapper
----

[[analysis]]
== Analysis Phase

The analyzer starts from the entry script and follows Lua module references.
It resolves candidates from:

* paths relative to the entry script;
* common project directories such as `src/`, `lib/`, and `lua_modules/`;
* `package.path`;
* `package.cpath`;
* explicit `--include` inputs.

Static analysis is intentionally conservative. Literal requires are supported:

[source,lua]
----
local model = require("model")
local cjson = require("cjson")
----

Dynamic requires need either runtime discovery or manual includes:

[source,lua]
----
local plugin = require("plugins." .. name)
----

[[trace]]
== Trace Records

Trace records explain why a module was selected, skipped, or reported as
missing. Use:

[source,sh]
----
luainstaller trace app/main.lua --max-deps 250
----

Trace output is useful before build because it shows:

* requiring file;
* source line where available;
* requested module name;
* candidate paths;
* selected path;
* selected type;
* reason;
* compatibility notes.

[[manifest]]
== Manifest Contract

Every bundle contains `.luai/manifest.lua`. It records the contract between
analysis, build, and runtime:

[cols="1,3",options="header"]
|===
|Manifest area |Purpose

|entry
|Original entry path and bundled destination.

|output
|Mode and output path.

|lua
|Lua version and Lua ABI note.

|platform
|Target OS and architecture.

|launcher
|Launcher profile and copied Lua runtime details.

|modules.lua
|Bundled Lua source modules.

|modules.native
|Copied native Lua C modules.

|trace
|Dependency resolution records.

|compatibility
|Notes and warnings about the selected profile.
|===

[TIP]
====
When a bundle runs but behaves differently from the source tree, inspect the
manifest first. It tells you which files the build believed were part of the
application.
====

[[launcher]]
== Launcher Generation

The launcher is generated C. It embeds a Lua bootstrap chunk and starts a Lua
state through the Lua C API.

At runtime the launcher:

[arabic]
. creates a Lua state;
. opens standard libraries;
. prepares the `arg` table;
. installs bundle-aware Lua and native module search paths;
. runs the embedded entry chunk;
. reports Lua errors with traceback context.

The generated source is kept under `.luai/build/launcher.c` in onedir bundles
for inspection.

[[onedir]]
== Onedir Assembly

An onedir bundle is the primary runtime layout:

[source,text]
----
app/
    app
    .luai/
        manifest.lua
        native/
            liblua5.4.so.0
            cjson.so
            socket/core.so
        build/
            launcher.c
----

Lua source modules are embedded into the launcher payload. Native modules are
copied as files because the platform dynamic loader needs real filesystem
paths.

[[onefile]]
== Onefile Assembly

Onefile builds reuse onedir. The process is:

[arabic]
. Build an inner onedir staging directory.
. Enumerate staged files.
. Generate an extractor C program.
. Embed staged files in the extractor.
. Compile the outer executable.
. At runtime, extract to a content-addressed cache and run the inner launcher.

[NOTE]
====
Existing extracted files may be reused when size and content hash match the
embedded payload.
====

[[runtime-search]]
== Runtime Search Paths

The bootstrap adjusts Lua loading so bundled modules are found before ambient
system modules.

For native modules, platform-specific search paths point at `.luai/native/`.
For Lua modules, the embedded searcher resolves from the bundled payload.

[[compatibility]]
== Compatibility Boundary

The first-stage promise is:

* same OS family;
* same architecture;
* same C ABI;
* same Lua ABI;
* compatible native libraries.

`luainstaller` does not claim that a bundle built on one arbitrary machine will
run on every machine with the same OS name.

[[comparison]]
== Relationship To PyInstaller-Style Packaging

Like PyInstaller, `luainstaller` analyzes an entry program, collects the
runtime needed by that program, and writes either a folder distribution or a
single-file wrapper. The implementation is Lua-specific:

[cols="1,2,2",options="header"]
|===
|Concern |PyInstaller-style concept |luainstaller concept

|Entry
|Python script
|Lua script

|Dependency discovery
|Python import graph
|Lua `require` graph and native Lua C modules

|Runtime
|Python interpreter
|Lua runtime selected by profile

|Folder output
|onedir
|onedir

|Single executable
|onefile extraction
|onefile extraction around an inner onedir layout

|Customization
|spec file and hooks
|CLI options, Lua API options, manual includes
|===

[[limitations]]
== Current Limitations

* General cross-compilation is not supported.
* Automatic external shared-library closure is not complete.
* Dynamic require discovery is path-sensitive.
* Native module availability must match the selected target profile.
* Windows bundles are built from a Linux host with MinGW.
