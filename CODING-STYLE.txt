luainstaller coding style
=========================

Scope
-----

These rules apply to project Lua, generated-C templates, shell tools, and tests.
Match the surrounding file when a rule does not settle a formatting choice.

Lua files
---------

* Target Lua 5.4 or newer. Do not add compatibility branches for older Lua
  releases unless the package dependency changes first.
* Indent with four spaces. Do not use tabs or trailing whitespace.
* Keep lines readable; split long command construction and error tables by
  meaning rather than at an arbitrary column.
* Use local variables and local helper functions by default. A module's public
  surface belongs on its returned `M` table.
* Name functions and variables in `lowerCamelCase` only where an existing module
  already does so; otherwise prefer `snake_case` for new local data. Do not
  rename stable public fields for style alone.
* Keep one responsibility per helper. Large parsing or build transactions
  should expose explicit boundaries and structured failure values.

Every tracked Lua file begins with the existing metadata block containing:

    Author:
    File:
    Date:
    Updated:

Update the date when a meaningful file change is made. Preserve the shebang
before the block in executable Lua scripts.

Errors and results
------------------

Public operations return one of these shapes:

    { ok = true, ... }

    {
        ok = false,
        error = {
            type = "SpecificErrorType",
            message = "actionable description",
            ...
        }
    }

Use the most specific existing error type. Include paths, option names, command
output, and restore state when they help the caller recover. Do not parse human
messages in tests when a structured field exists.

Internal code may throw an error table when unwinding several parser frames.
Convert it at the module boundary with `result.fromThrown`. Avoid throwing plain
strings from public orchestration code.

Paths and processes
-------------------

* Normalize project paths through `luainstaller.path`.
* Quote every path or argument passed to a shell through
  `luainstaller.process.shellQuote`.
* Never concatenate unquoted command-line, environment, manifest, or trace data
  into a shell command.
* Treat output deletion and temporary directories as security boundaries.
* Build replaceable output in staging and commit it only after every fallible
  generation step succeeds.
* Reject symlinks where a path is expected to remain inside an owned output or
  extraction root.
* Check write, close, copy, compiler, and rename results when their failure can
  change the public result.

Generated C
-----------

* Keep the standalone launcher template and the embedded default synchronized.
* Use bounded writes and check every buffer length.
* Preserve platform branches for Linux, macOS, and Windows.
* Follow the target runtime's argument quoting rules; POSIX and Windows command
  lines are not interchangeable.
* Any extractor path must remain below its private root after normalization.
* Compile generated C on each affected target. Static inspection is not enough.

Shell tools
-----------

* Use POSIX `sh` unless a script explicitly requires Bash.
* Start release/test tools with `set -eu`.
* Quote variable expansions except in the few places where word splitting is the
  documented data format.
* Keep credentials in environment variables. Never print or commit them.
* Use `/tmp` prefixes for remote test installations and clean only paths owned by
  that script.
* A remote test must fail when required coverage is skipped. Print an explicit
  skip only for an optional local diagnostic.

Tests
-----

* Reproduce a defect with a failing test before changing production code.
* Assert behavior visible to callers, generated files, or target runtimes.
* Keep fixtures minimal and give failures enough context to diagnose the layer.
* Clear `LUA_PATH` and `LUA_CPATH` when testing bundle independence.
* Do not accept a toolchain error as a successful architecture test.
* Use `test/support/` for shared mechanics.
* Preserve user files and unrelated worktree changes.

Documentation
-------------

* Write for a concrete task. Put a fact in one primary guide and link to it from
  other guides.
* Prefer a short example plus an explanation over repeated option tables.
* State compatibility limits plainly. Do not imply universal portability.
* Every documented command must be executable from the directory it names.
* Update `test/contract_docs.lua` when a public promise changes.

Required checks
---------------

    find src test -type f -name '*.lua' -print0 | xargs -0 -n1 luac -p
    find tools -type f -name '*.sh' -print0 | xargs -0 -n1 sh -n
    git diff --check
    lua test/cli_split_smoke.lua
    lua test/contract_docs.lua
    lua test/smoke_all.lua

Generated-C, native-module, or platform changes also require the applicable
Linux x86_64, Linux ARM64, macOS ARM64, Wine, and Windows VM checks described in
docs/TESTING.adoc.
