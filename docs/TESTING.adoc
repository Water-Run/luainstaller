This guide defines the checks required before a release. All release targets
are physical native hosts; the project does not use a VM, Wine, MinGW, or a
cross-compiler as release evidence.

toc::[]

[#testing-local]
== Local contract suites

Run from the repository root with the official Lua interpreter under test:

[source,sh]
----
LUA=${LUA:-lua}
"$LUA" test/cli_split_smoke.lua
"$LUA" test/contract_docs.lua
"$LUA" test/production_edges.lua
"$LUA" test/smoke_all.lua
luarocks lint luainstaller-1.0.0-1.rockspec
----

`cli_split_smoke.lua`::
Checks that `luai` accepts only its flag grammar and `luainstaller` accepts only
its descriptive grammar.

`contract_docs.lua`::
Turns public documentation promises into executable contracts.

`production_edges.lua`::
Exercises fail-closed filesystem handling, checked I/O, source snapshots,
SHA-256 ownership, output transactions, runtime tracing, logs, remote-script
safety, and deterministic onefile behavior.

`smoke_all.lua`::
Runs the broad API, sample, dependency-discovery, generated-C, onedir, and
onefile suite.

Tests under `test/` may use syntax from the interpreter selected for that
particular suite. All production files under `src/` must parse on every
supported Lua major/minor version.

[#testing-luarocks]
== Installation contract

LuaRocks is the only supported installation path:

[source,sh]
----
luarocks install luainstaller
luai -v
luainstaller version
----

The release test installs into an isolated tree, invokes both installed command
names from outside the checkout, builds an outside-checkout application, then
runs the generated executable with `LUA_PATH`, `LUA_CPATH`, and Lua-bearing
`PATH` entries cleared. A missing LuaRocks command is a failure, not a skip.

[#testing-version-matrix]
== Official Lua 5.x matrix

The pinned matrix covers these upstream patch releases:

[cols="1,2",options="header"]
|===
|ABI |Tested patch release
|Lua 5.1 |5.1.5
|Lua 5.2 |5.2.4
|Lua 5.3 |5.3.6
|Lua 5.4 |5.4.8
|Lua 5.5 |5.5.0
|===

The runners verify SHA-256 before extracting any source archive. Per version,
they check production-source syntax, version rejection (including LuaJIT), CLI
behavior, LuaRocks installation, native toolchain ABI, onedir, onefile,
reproducibility, and clean-target execution.

On Linux or macOS:

[source,sh]
----
HOST_LABEL=my-physical-host sh tools/test-lua-versions.sh
----

On Windows:

[source,powershell]
----
powershell.exe -NoLogo -NoProfile -NonInteractive `
  -ExecutionPolicy Bypass -File .\tools\test-lua-versions.ps1 `
  -HostLabel my-physical-windows-host
----

`VersionFilter` may shorten a diagnostic rerun, but a release requires all five
pins. Evidence is written as one log per host and Lua patch version below a
guarded `luainstaller-*` temporary directory.

[#testing-orchestration]
== Lab orchestration

Linux physical hosts:

[source,sh]
----
sh tools/remote-test-linux.sh
----

macOS physical hosts (run once for each alias):

[source,sh]
----
MAC_HOST=yymac06 sh tools/remote-test-macos.sh
MAC_HOST=yymacstudio sh tools/remote-test-macos.sh
----

Windows runs directly on the Windows machine with
`tools/test-lua-versions.ps1`. `tools/remote-test-windows.sh` is only a small
compatibility launcher for a POSIX environment that already provides `pwsh`;
it contains no SSH or cross-build workflow.

Linux and macOS orchestration transfer `git archive --format=tar HEAD`, not an
untracked working tree. Temporary and cache roots must be normalized children
of `/tmp/luainstaller-*`, and archive downloads must match their pinned digest.

[#testing-release]
== Release checklist

. LuaRocks lint passes and installation from the rockspec works in an isolated
  tree.
. Production Lua source parses on every pinned official Lua version.
. Full local contract suites pass without a required skip.
. All five Lua versions pass on every listed physical host.
. Each native onedir and onefile output runs with no Lua command or Lua module
  paths available on the target.
. Repeated onefile builds and repeated extraction checks pass.
. Copied native modules and external system libraries are inspected on their
  actual target.
. Documentation contains no instruction that requires source installation,
  cross-building, Wine, a VM, MinGW, or a device allowlist.
. `git diff --check` and the final clean-checkout verification pass before the
  release commit is pushed.
