[#usage-workflow]
== Use the source program as the baseline

Before packaging, run the entry script with the same Lua interpreter and module
paths that will be used for analysis:

[source,sh]
----
lua app/main.lua --help
----

Then work in this order:

. analyze dependencies;
. inspect the trace when a resolution is surprising;
. build an onedir bundle;
. run it with `LUA_PATH` and `LUA_CPATH` cleared;
. build onefile only after onedir works.

This order separates dependency errors from compiler, native loader, and
onefile extraction errors.

[#usage-commands]
== Two command surfaces

The short and descriptive interfaces are equivalent, but their grammar is not
interchangeable.

[source,sh]
----
luai -a app/main.lua
luai -t app/main.lua
luai -b --dir app/main.lua -o build/app

luainstaller analyze app/main.lua
luainstaller trace app/main.lua
luainstaller build --dir app/main.lua -o build/app
----

Use `luai -h` or `luainstaller help` to print the accepted form. A parse error
returns exit status 1 and writes a short hint to standard error.

[#usage-first-bundle]
== Build and inspect an onedir bundle

[source,sh]
----
luainstaller trace test/runtime_bundle/main.lua --max-deps 120
luainstaller build --dir test/runtime_bundle/main.lua \
  -o build/runtime-demo --max-deps 120

env -i PATH=/usr/bin:/bin \
  build/runtime-demo/runtime-demo Grace
----

The clean environment is important: it prevents a system `LUA_PATH` or
`LUA_CPATH` from hiding a missing bundled dependency.

Inspect the generated files before distribution:

[source,sh]
----
find build/runtime-demo -maxdepth 4 -type f | sort
sed -n '1,220p' build/runtime-demo/.luai/manifest.lua
----

The launcher contains the entry script and pure-Lua modules. `.luai/native/`
contains copied Lua C modules and, on shared-Lua profiles, the Lua runtime.

[#usage-onefile]
== Build onefile

[source,sh]
----
luainstaller build --file test/runtime_bundle/main.lua \
  -o build/runtime-demo-onefile --max-deps 120
build/runtime-demo-onefile Grace
----

At runtime, onefile extracts a content-addressed onedir payload below the
platform temporary directory and starts its inner launcher. POSIX caches are
isolated by effective user ID and require owner-only permissions. Existing
files are reused only when their size and hash match.

[#usage-options]
== Options

=== Output selection

`--dir`, `--onedir`::
Build a directory bundle. This is the default.

`--file`, `--onefile`::
Build a self-extracting executable.

`-o PATH`, `--out PATH`::
Set the output directory for onedir or executable path for onefile. Existing
user-owned content is never overwritten. A recognized generated onedir may be
rebuilt through a private sibling staging directory. The final directory swap
has a brief path-visibility gap and is not crash-atomic.

=== Dependency inputs

`--include PATH`::
Add a Lua source file that automatic discovery did not find. Repeat the option
for multiple files.

`--exclude VALUE`::
Remove a resolved dependency. `VALUE` may be an exact path, a basename, or a
path suffix beginning on a `/` boundary. Excludes take precedence over
automatic discovery and manual includes.

`-d MODE`, `--discovery-mode MODE`::
Choose `static`, `runtime`, or `manual`.

`--no-depscan`::
Disable automatic discovery. Equivalent to manual mode.

`--max-deps N`::
Limit discovered Lua source dependencies. The default is 36. Larger
applications should set an explicit, reviewed limit.

=== Runtime discovery

`--lua PATH`::
Select the interpreter used to execute the tracer. Resolution order is this
option, `LUAI_LUA`, the current interpreter, then `lua` from `PATH`.

`-- ARGS...`::
Pass arguments to the entry script during runtime discovery. These arguments
are not saved as defaults for the finished executable.

=== Target profile

`--target-os OS`::
Select `linux`, `macos`, or `windows`. This selects a build profile; it does not
convert native machine code.

`--lua-prefix PATH`::
Provide the Lua headers and runtime required by macOS and Windows profiles.
`LUAI_LUA_PREFIX` is used when the option is absent.

`--verbose`::
Print additional trace or manifest counts.

[#usage-discovery]
== Choose a discovery mode

=== Static

Static mode is the default. It parses literal forms such as:

[source,lua]
----
local json = require("cjson")
local socket = require 'socket'
local optional_ok, optional = pcall(require, "optional_plugin")
----

Lua string escapes are decoded before module resolution. The common same-name
binding `local require = require` is accepted; an alias such as
`local load_module = require` is not statically followed.

Computed module names return `DynamicRequireError`:

[source,lua]
----
require(prefix .. ".adapter")
----

Use runtime mode for a computed name that can safely execute at build time, or
manual mode plus `--include` when execution is unsafe.

=== Runtime

Runtime mode runs the entry under a temporary `require` tracer:

[source,sh]
----
luai -a app/main.lua -d runtime -- profile_a input.json
----

Only code paths executed by that run are observed. Successful module loads are
recorded; missing modules caught by the application's `pcall` remain optional.
The tracer flushes records before `os.exit`.

Do not use runtime discovery for an entry that performs destructive work at
startup. It is execution, not static inspection.

=== Manual

Manual mode starts with no discovered dependencies:

[source,sh]
----
luai -b app/main.lua --no-depscan \
  --include app/core.lua \
  --include app/adapters/sqlite.lua \
  -o build/app
----

Manual includes need to preserve their module path relative to the entry
directory. For example, include `app/adapters/sqlite.lua`, not a flattened copy
named `sqlite.lua`.

[#usage-native]
== Native modules

Trace and build a native application with the same `LUA_PATH` and `LUA_CPATH`
used by the source program:

[source,sh]
----
LUA_PATH='/opt/app/share/lua/5.4/?.lua;;' \
LUA_CPATH='/opt/app/lib/lua/5.4/?.so;;' \
luainstaller build --dir app/main.lua -o build/app --max-deps 250
----

After building:

[source,sh]
----
find build/app/.luai/native -type f | sort
ldd build/app/app
find build/app/.luai/native -type f -name '*.so' -exec ldd {} \;
----

Replace `ldd` with `otool -L` on macOS. Test Windows output on Windows even if
it already passed under Wine.

[#usage-logs]
== Logs

The descriptive CLI stores failure records below `~/.luainstaller/`:

[source,sh]
----
luainstaller logs
luainstaller logs --level error --limit 20
luainstaller logs --asc
----

The log is diagnostic context, not a build transcript. Compiler output and
structured error details returned by the current command remain the primary
evidence.

[#usage-api]
== Lua API

[source,lua]
----
local luainstaller = require("luainstaller")

local analysis = luainstaller.analyze({
    entry = "app/main.lua",
    discovery_mode = "static",
    max_deps = 120,
})

if not analysis.ok then
    io.stderr:write(analysis.error.type, ": ", analysis.error.message, "\n")
    os.exit(1)
end

local built = luainstaller.bundle({
    entry = "app/main.lua",
    mode = "onedir",
    out = "build/app",
    max_deps = 120,
})
assert(built.ok, built.error and built.error.message)
print(built.executable)
----

Public operations return `{ ok = true, ... }` or
`{ ok = false, error = { type = ..., message = ... } }`. Invalid or unknown
option keys return `InvalidOptionsError` rather than being ignored.

Available operations are `analyze`, `trace`, `compatibility`, `bundle`,
`getLogs`, and `clearLogs`.
