This guide explains the platform model behind `luainstaller`: what a
target profile means, how native Lua C modules are copied, which
external dependencies remain outside the bundle, and which portability
claims the project intentionally does not make.


toc::[]

[#platforms-compatibility]
== Compatibility Model

The core rule is simple but strict:

[IMPORTANT]
====
A bundle is expected to run on a target with the same OS family, CPU
architecture, system ABI, and Lua ABI as the selected build profile.
====

This is narrower than "cross-platform executable". The generated
launcher can be built for different profiles, but native Lua C modules,
Lua runtime libraries, libc families, compiler runtimes, and third-party
shared libraries still have platform rules that `luainstaller` does not
rewrite.

[#platforms-targets]
== Target Profiles

[cols="1,2,3,3",options="header"]
|===
|Target |Select with |Build prerequisites |Runtime strategy

|Linux
|Default on Linux or `--target-os linux`
|Linux host, `cc`, `pkg-config --cflags --libs lua`, Lua headers, and
native modules built for the same Lua ABI.
|Shared-Lua launcher with `$ORIGIN/.luai/native`; the linked
`liblua*.so` is copied into `.luai/native/`.

|macOS
|Default on macOS or `--target-os macos`
|macOS host, `cc`, and a Lua prefix containing `include/lua.h` and
`lib/liblua.a`.
|Static Lua launcher using `liblua.a`; native module paths are relative
to `@loader_path/.luai/native`.

|Windows
|`--target-os windows` from Linux
|Linux host, MinGW compiler, a Windows Lua prefix containing
`include/lua.h` and `lua54.dll` or `lua.dll`, plus Windows builds of
required native modules.
|MinGW-built `.exe`; the Lua DLL is copied beside the launcher and into
`.luai/native/`.
|===

Environment variables:

[cols="1,2,3",options="header"]
|===
|Variable |Used by |Effect

|`LUAI_TARGET_OS`
|API and CLI bundle path when `--target-os` is omitted.
|Selects `linux`, `macos`, or `windows`.

|`LUAI_LUA_PREFIX`
|macOS and Windows profiles.
|Default Lua prefix when `--lua-prefix` is omitted.

|`LUAI_WINDOWS_CC`
|Windows profile.
|Overrides `x86_64-w64-mingw32-gcc`.

|`LUAI_LUA`
|Source-installed wrapper scripts.
|Overrides the Lua interpreter used by installed wrappers.

|`PREFIX`, `LUA`
|`tools/install-source.sh`.
|Default source install prefix and Lua command.
|===

[#platforms-linux]
== Linux Profile

Linux onedir output uses a shared-Lua launcher. The build relies on the
host's compiler and Lua development metadata:

[source,sh]
----
cc --version
pkg-config --cflags --libs lua
----

The launcher compile command is shaped like this:

[source,text]
----
cc .luai/build/launcher.c -o <exe> -Wl,-rpath,'$ORIGIN/.luai/native' $(pkg-config --cflags --libs lua)
----

After compilation, `ldd` is used to locate the linked Lua shared
runtime. That runtime is copied into `.luai/native/` and recorded in the
manifest. This helps a bundle run on a matching environment without
requiring a separately installed `lua` command.

Linux portability remains ABI-bound. A bundle built against glibc is not
made musl-compatible by packaging. A native module built for one
architecture is not translated to another architecture.

[#platforms-macos]
== macOS Profile

The macOS profile expects a Lua prefix with a static Lua library:

[source,text]
----
<prefix>/include/lua.h
<prefix>/lib/liblua.a
----

The static Lua runtime simplifies distribution of the launcher itself,
but native modules still have their own loader dependencies. Frameworks,
`.dylib` dependencies, deployment targets, codesigning, notarization,
and `.app` bundle layout are outside the current scope.

[#platforms-windows]
== Windows Profile

The Windows profile is currently built from Linux with MinGW. The Lua
prefix must be for Windows, not the Linux host:

[source,sh]
----
x86_64-w64-mingw32-gcc --version
test -f "$LUAI_LUA_PREFIX/include/lua.h"
test -f "$LUAI_LUA_PREFIX/bin/lua54.dll" || test -f "$LUAI_LUA_PREFIX/lua54.dll"
----

Native Lua C modules must also be Windows builds. Cross-compiling the
launcher does not turn Linux `.so` files into Windows `.dll` files.

[#platforms-native]
== Native Lua C Modules

Lua C modules are files loaded by Lua's native loader:

[source,lua]
----
require("cjson")
require("lsqlite3")
require("socket.core")
require("lfs")
----

The bundler copies detected native module files into `.luai/native/`.
For dotted names it writes a nested path when the trace identifies the
requested module:

[source,text]
----
require("socket.core")
  -> .luai/native/core.so
  -> .luai/native/socket/core.so
----

The copied file must already be correct for the target OS, CPU
architecture, Lua ABI, and compiler/runtime family. `luainstaller`
preserves and lays out native modules; it does not rebuild them for a
new platform.

Static archives such as `.a` are not Lua `require()` native modules and
are not discovered as dependencies. They are used only by target profiles
that explicitly link a launcher against a static Lua runtime, such as the
macOS `liblua.a` profile.

[#platforms-external]
== External Library Boundary

A native Lua module can depend on libraries that are not Lua modules:

* `libsqlite3.so.0`;
* `libssl.so.3`;
* C++ runtime libraries;
* system frameworks;
* third-party DLLs.

Inspect native modules directly:

[source,sh]
----
ldd dist/app/.luai/native/lsqlite3.so
otool -L dist/app/.luai/native/lsqlite3.so
----

[WARNING]
====
`luainstaller` copies detected Lua C module files and selected Lua
runtime files. It does not automatically compute and copy the full
external shared-library dependency closure of every native module.
====

[#platforms-verification]
== Verified Matrix

The current local and lab verification loop covers:

* Linux x86_64 full smoke, LuaRocks install, source install, and clean
  runtime bundle execution;
* Linux aarch64 source install and analysis, with a clean
  `ToolchainError` when Lua headers and Lua `pkg-config` metadata are
  unavailable;
* macOS arm64 onedir bundles for selected pure Lua and native samples,
  plus pure Lua and student onefile bundles;
* Windows 10 x64 bundles built from Linux with MinGW and a Windows Lua
  prefix, then run without installed Lua or LuaRocks on the target.

The matrix is evidence for the sample set, not a promise that arbitrary
native packages or OS distributions will work without target-specific
inspection.

[#platforms-limits]
== Boundaries and Non-Goals

Not promised:

* universal cross-platform output;
* cross-distribution Linux portability;
* glibc/musl compatibility;
* architecture translation;
* native module ABI conversion;
* full external shared-library dependency closure;
* no-extract onefile native loading;
* codesigning, notarization, app bundles, MSI/PKG/DEB/RPM installers;
* sandboxing or permission isolation.

[#platforms-security]
== Security Notes

`luainstaller` is a packaging tool, not a sandbox:

* bundled Lua code runs with the process privileges of the executable;
* manifest hashes identify packaged bytes but do not establish trust;
* onefile payloads are embedded, not encrypted;
* onefile extraction writes files to a temp/cache directory;
* output safety guards protect build outputs from accidental overwrite,
  not runtime application behavior.
