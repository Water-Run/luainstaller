[#bundling-onedir]
== Directory layout

Linux / macOS:

[source,text]
----
app/
├── app
└── .luai/
    ├── build/
    │   └── launcher.c
    ├── generated-output.txt
    ├── manifest.lua
    └── native/
        ├── liblua-5.x.so    # shared-Lua profiles; name follows ABI
        └── module.so
----

Windows places the Lua DLL beside the executable and a second copy under
`.luai/native/` for dependent modules.

The entry and pure-Lua modules are compiled into the C launcher and listed in
the manifest; they are not left as loose `.lua` files in the tree.

[#bundling-build]
== Build placement

Generation, copy, compile, and manifest write happen in a private staging
directory, then rename into the final path. When replacing a generated bundle,
the old tree is backed up and revalidated; on failure the previous bundle is
restored when possible.

The output lock only coordinates `luainstaller` itself. On POSIX, another
process holding a writable descriptor opened before rename may still mutate
that inode after verification. Use a new output path when that cannot be
ruled out.

[#bundling-launcher]
== Startup

. Create a Lua state and open standard libraries
. Set `arg`
. Load the embedded bootstrap as text-only Lua
. Run the entry with a traceback handler

The process image path comes from the OS (not the spelling of `argv[0]`), so
launch via `PATH` or a symlink still locates `.luai/native`.

[#bundling-lua]
== Pure-Lua lookup order

. `package.preload`
. Modules embedded in the bundle
. Filesystem Lua searchers
. Native searchers

Packaged modules take precedence without a hermetic sandbox: undeclared names
may fall through to the host. Embedded code uses `load(..., "t")` (no bytecode
payload). Name collisions → `DuplicateModuleError`.

[#bundling-native]
== Native modules

Copied by basename and, when known, dotted path—for example `socket.core`:

[source,text]
----
.luai/native/core.so
.luai/native/socket/core.so
----

`package.cpath` is prepended with `.luai/native` patterns for `?.so` /
`?.dylib` / `?.dll` and `?/init.*`. RPATH covers the copied Lua runtime;
further system libraries of a C module are not auto-packaged. Before publish, a
native linked probe requires `_VERSION` to match the build interpreter.

[#bundling-manifest]
== Manifest and ownership

`.luai/manifest.lua` records entry, hashes, mode, Lua ABI, platform, module
groups, include/exclude, and trace data. Format `version = 2` with
`hash_algorithm = "sha256"`. Sources are re-read before embed/copy; mismatch →
`SourceChangedError`.

`.luai/generated-output.txt` is the v2 ownership marker
(`luainstaller-generated-output-v2`): final directory, every generated
directory, and every generated regular file with SHA-256. Replacement is
allowed only when the **recursive** inventory still describes the
**exact file set** (hashes included). Tree edits, symlinks, or record mismatch refuse
automatic overwrite. Markers from `luainstaller-generated-output-v1` are never
auto-upgraded; migration is a **one-time removal** after backup, or a new
output path.

[#bundling-onefile]
== Single-file

Build a full directory bundle first, then embed runtime files into the
extractor (path, size, executable bit, bytes). `.luai/build/` and the onedir
ownership marker are omitted. An existing output path is not overwritten.

POSIX cache: `luainstaller-onefile-<effective-uid>` under the canonical
temporary root; permission and sticky-ancestor rules are in
xref:TROUBLESHOOTING.adoc[Troubleshooting]. Windows uses the user temporary
directory with reparse/DACL constraints.

Cache hits require full-byte equality. After confirming no process uses the
cache:

[source,bash]
----
rm -rf "${TMPDIR:-/tmp}/luainstaller-onefile-$(id -u)"
----

If single-file fails, verify the matching directory bundle for the same entry
first.
