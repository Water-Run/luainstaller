[#bundling-onedir]
== Onedir layout

A Linux or macOS onedir bundle has this shape:

[source,text]
----
app/
├── app
└── .luai/
    ├── build/
    │   └── launcher.c
    ├── generated-output.txt
    ├── manifest.lua
    └── native/
        ├── liblua-5.4.so        # shared-Lua profiles
        └── module.so            # when required
----

Windows also places the Lua DLL beside the executable so the Windows loader can
start the launcher. A second copy lives in `.luai/native/` for dependent native
modules.

The entry script and pure-Lua dependencies are encoded into the generated C
launcher. They are listed in the manifest but are not copied as loose source
files.

[#bundling-build]
== Build transaction

`bundleOnedir` validates the target profile and final output before creating any
generated files. It then builds in a private sibling staging directory:

. generate the Lua bootstrap and C launcher;
. copy Lua C modules;
. compile the launcher;
. copy the shared Lua runtime where required;
. write `manifest.lua` and the generated-output marker;
. rename the completed staging directory into place.

When replacing an existing generated bundle, the old directory is renamed to a
backup immediately before the final switch. If the switch fails, the backup is
restored. A compiler or copy failure therefore leaves the previous bundle
available instead of replacing it with partial output.

[#bundling-launcher]
== Launcher startup

The native launcher performs four tasks:

. create a Lua state and open the standard libraries;
. set `arg` from the process arguments;
. load the embedded bootstrap as text-only Lua code;
. execute the entry with a traceback handler.

The launcher obtains its process image path from the operating system rather
than trusting the spelling of `argv[0]`. Native modules therefore remain
locatable when the executable is reached through `PATH` or a symbolic link.

Platform path sources are `/proc/self/exe` on Linux,
`_NSGetExecutablePath` plus `realpath` on macOS, and `GetModuleFileNameA` on
Windows.

[#bundling-lua]
== Pure-Lua module loading

The bootstrap installs a package searcher at position 2, immediately after
`package.preload`. Lookup order is therefore:

. application or host `package.preload` entry;
. Lua module embedded in the bundle;
. normal filesystem Lua searchers;
. native module searchers.

An embedded module is compiled with `load(..., "t")`, so bytecode is not
accepted through the generated source payload. A UTF-8 BOM and a leading
shebang are stripped before loading.

Module names come from dependency trace records. If two source files map to the
same module name or manifest destination, the build stops with
`DuplicateModuleError`.

[#bundling-native]
== Native module loading

Native modules are copied under both their file basename and, when known, their
dotted module path. For `socket.core`, for example, the bundle can contain:

[source,text]
----
.luai/native/core.so
.luai/native/socket/core.so
----

The bootstrap prepends these patterns to `package.cpath`:

[source,text]
----
.luai/native/?.so
.luai/native/?/init.so
.luai/native/?.dylib
.luai/native/?/init.dylib
.luai/native/?.dll
.luai/native/?/init.dll
----

The launcher RPATH handles the copied Lua runtime. It does not copy every shared
library referenced by a Lua C module. Those external dependencies remain a
release responsibility.

[#bundling-manifest]
== Manifest

`.luai/manifest.lua` is readable Lua data prefixed with
`-- generated by luainstaller`. It records:

* entry source and content hash;
* output mode and requested path;
* Lua version and ABI;
* target platform and launcher profile;
* Lua, native, and external module groups;
* manual include/exclude settings;
* dependency trace records;
* the packaged Lua runtime.

Hashes use 32-bit FNV-1a. They support generated-output verification and cache
reuse; they are not cryptographic signatures.

[#bundling-marker]
== Output ownership marker

`.luai/generated-output.txt` contains:

* marker version `luainstaller-generated-output-v1`;
* normalized final output directory;
* expected top-level files and their hashes.

An existing directory is replaceable only when all of these checks pass:

* it is empty, or it contains a generated manifest and valid marker;
* it is not a symbolic link;
* every top-level entry is expected;
* every marked top-level file still has the recorded hash;
* the marker names the same output directory.

Editing or adding a top-level file intentionally revokes automatic replacement.
Choose a new output path or remove the directory after reviewing its contents.

[#bundling-onefile-build]
== Onefile construction

Onefile first builds a normal onedir bundle in a private temporary location.
Every staged regular file is then embedded in a small extractor with its path,
size, executable bit, and FNV-1a hash. The payload ID is derived from the sorted
file paths and hashes.

The final extractor is compiled for the selected target. Onefile output refuses
to replace any existing file or directory; remove or rename an old artifact
explicitly before rebuilding it. Publishing uses a same-filesystem hard link so
the final path is created atomically without replacing a file that appeared
during compilation. FAT/exFAT mounts and some network filesystems do not support
this operation; choose a native filesystem output directory when the build
returns `FilesystemError` for atomic publication.

[#bundling-onefile-runtime]
== Onefile extraction and cache

On POSIX, the cache directory is named `luainstaller-onefile-<effective-uid>`
under the canonical temporary root. Both the cache and payload directory must:

* be owned by the effective user;
* deny all group and other permissions;
* be real directories, not symbolic links.

Windows uses the current user's temporary directory, rejects reparse points,
verifies that cache directories are owned by the current process token's user
SID, and applies a protected DACL granting full access only to that user and
`SYSTEM`. The extractor reapplies the same owner and protected DACL to both
new and hash-matching cached files, including caches created by older releases.

Before writing any file, the extractor rejects absolute paths, drive-qualified
paths, empty segments, `.` segments, and `..` segments. Existing final-file
symlinks or reparse points are removed rather than followed. Parent directory
symlinks are rejected. A cached file is reused only when size and hash match;
executable mode is repaired on POSIX when necessary.

The cache is persistent. Remove it to force extraction after checking that no
bundle process is using it:

[source,sh]
----
rm -rf "${TMPDIR:-/tmp}/luainstaller-onefile-$(id -u)"
----

[#bundling-debug]
== Inspect onedir before onefile

If onefile fails, rebuild the same entry as onedir and check:

[source,sh]
----
find build/app -maxdepth 5 -type f | sort
sed -n '1,240p' build/app/.luai/manifest.lua
env -i PATH=/usr/bin:/bin build/app/app
----

If onedir works but onefile does not, investigate temporary-directory access,
cache ownership, path length, endpoint security software, and child-process
creation. Do not begin with dependency discovery; onefile embeds the already
built onedir result.
