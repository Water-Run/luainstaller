[#implementation-flow]
== End-to-End Flow

Every public operation starts from an entry script and a normalized
option table. Analysis and trace stop after dependency discovery;
bundling continues through manifest creation and launcher generation.

[source,text]
----
CLI or Lua API
  -> normalize options
  -> choose dependency engine
  -> produce dependency plan
  -> build manifest
  -> generate Lua bootstrap
  -> generate and compile C launcher
  -> assemble onedir output
  -> optionally wrap as onefile
----

The important distinction is that dependency discovery and bundle
assembly are separate phases. A dependency plan says what should be
packaged. The bundler decides how those files are laid out for a target
profile.

[#implementation-entry-points]
== Public Entry Points

[cols="1,3",options="header"]
|===
|Entry point |Responsibility

|`luainstaller.analyze(opts)`
|Normalize options, run the selected dependency engine, and return Lua
source/native module lists plus trace records.

|`luainstaller.trace(opts)`
|Run dependency discovery and add compatibility diagnostics for the
selected target profile.

|`luainstaller.compatibility(opts)`
|Report host, target, Lua ABI, mode, native-module count, notes, and
warnings without building a bundle.

|`luainstaller.bundle(opts)`
|Run discovery, build a manifest, and dispatch to onedir or onefile
assembly.

|`src/cli.lua`
|Parse compact `luai` syntax and full-word `luainstaller` syntax, then
render stable command-line output.
|===

Public functions return result tables instead of throwing for normal
user mistakes:

[source,lua]
----
{
  ok = false,
  action = "bundle",
  error = {
    type = "ToolchainError",
    message = "...",
    details = { ... },
  },
}
----

Internal modules may throw structured tables while doing file or
compiler work. `src/init.lua` converts those into public result tables
at the API boundary.

[#implementation-boundaries]
== Module Boundaries

[cols="1,2,3",options="header"]
|===
|Module |Owns |Does not own

|`src/init.lua`
|Public API shape, option normalization, dependency-plan dispatch,
result conversion.
|CLI rendering, compiler command construction, persistent log storage.

|`src/cli.lua`
|Command grammar, split `luai` / `luainstaller` personalities, text
rendering, log command options.
|Dependency resolution rules or bundle layout decisions.

|`src/analyzer.lua`
|Static require extraction, Lua/native candidate resolution, recursive
source scanning, trace records.
|Runtime execution of user code.

|`src/require_engine.lua`
|Choosing static/manual/runtime discovery and applying include/exclude
rules.
|Manifest schema and launcher compilation.

|`src/manifest.lua`
|Schema version 1 manifest, normalized file records, hashes, duplicate
destination checks.
|Copying files into the output directory.

|`src/cgen.lua`
|Lua bootstrap generation and embedded Lua payload table.
|Native module copying or C compiler invocation.

|`src/launcher.lua`
|C launcher source generation for the embedded bootstrap.
|Target-specific output validation.

|`src/bundler.lua`
|Onedir validation, target profile checks, native/runtime file copying,
launcher compilation, final manifest write.
|Onefile extractor generation.

|`src/onefile.lua`
|Temporary onedir staging, file embedding, extractor C generation,
outer executable compilation.
|Dependency discovery.

|`src/platform.lua`
|Host detection and target profile facts: suffixes, native extensions,
runtime paths, Lua prefix.
|Compiler output parsing.

|`src/compat.lua`
|Compatibility summary, target notes, native-module warnings.
|Enforcing compatibility at runtime.

|`src/logger.lua`
|Persistent logs under `~/.luainstaller`.
|The generated bundle manifest.
|===

[#implementation-discovery]
== Dependency Discovery

The default static engine is a require-focused lexer, not a full Lua
parser. That is deliberate: it keeps discovery predictable and avoids
executing user code during the normal path.

Recognized forms include literal requires and optional probes:

[source,lua]
----
require("model")
require 'storage'
require([[reports]])
pcall(require, "optional.module")
----

Rejected forms are dynamic because the module name is not known from the
source text:

[source,lua]
----
local name = "model"
require(name)
require("plugin." .. kind)
----

Resolution follows Lua's search model closely enough for packaging:

[arabic]
. Ignore known Lua built-in modules.
. Expand the requested name through `package.path`.
. Expand the requested name through `package.cpath`.
. Record every candidate path considered.
. Classify the selected file as Lua source or native module.
. Recurse into Lua source dependencies.

Trace records are not just debug strings. They are structured data used
by CLI rendering, manifests, tests, and native-module destination logic.
Important fields include `requiring_file`, `source_line`, `requested`,
`optional`, `candidates`, `selected_path`, `selected_type`,
`classification`, and `reason`.

[#implementation-engines]
== Discovery Engines

[cols="1,3",options="header"]
|===
|Engine |Behavior

|`static`
|Run the analyzer over literal require calls, resolve through Lua and
native search paths, and recursively scan Lua source files.

|`manual`
|Start from an empty automatic plan, add explicit `include` files, then
apply excludes. Manual includes are treated as Lua source files.

|`runtime`
|Write a temporary tracer, wrap `require`, run the entry script once,
record module names that were actually loaded, and resolve those
records.
|===

Runtime discovery is intentionally limited. It only sees the code path
executed during the build-time run. If an application loads modules
behind different subcommands, the caller must pass representative
arguments after `--` or add manual includes.

[#implementation-manifest]
== Manifest Contract

`.luai/manifest.lua` is executable Lua data so users can inspect it with
plain Lua tools:

[source,lua]
----
-- generated by luainstaller
return {
  version = 1,
  hash_algorithm = "fnv1a32",
  entry = { ... },
  output = { ... },
  lua = { ... },
  platform = { ... },
  launcher = { ... },
  modules = {
    lua = { ... },
    native = { ... },
    external = {},
  },
  manual = { ... },
  trace = { ... },
  compatibility = { ... },
}
----

The manifest is written after the launcher and runtime files have been
assembled, because it records final output paths, runtime records, and
compatibility notes. It starts with `-- generated by luainstaller`; the
bundler uses that marker when deciding whether an existing output
directory is safe to replace.

Manifest invariants:

* destination paths are normalized;
* Lua dependency manifest destinations preserve relative paths under
  `.luai/lua/`, even though source bytes are embedded into the launcher
  instead of copied as ordinary files in the current onedir layout;
* native dependencies are recorded under `.luai/native/`;
* duplicate destinations fail with `DuplicateModuleError`;
* file hashes use the manifest's declared `fnv1a32` algorithm;
* trace and compatibility data stay with the generated artifact.

[#implementation-runtime]
== Runtime Generation

The generated executable embeds Lua source in a bootstrap rather than
requiring the original source tree at runtime.

[source,text]
----
cgen.buildPayload
  -> read entry source and Lua dependency source
  -> map module names to source records
  -> emit Lua bootstrap source

launcher.generateSource
  -> convert bootstrap source to a C byte array
  -> append the Lua C API launcher template
----

At runtime the C launcher:

[arabic]
. creates a Lua state;
. opens the standard libraries;
. builds `arg`;
. installs traceback handling;
. loads the embedded bootstrap;
. calls the bootstrap.

The bootstrap configures native module search paths relative to the
launcher, installs a bundled Lua source searcher at searcher position 2,
strips BOM and shebang lines from bundled source, and runs the entry
chunk.

[#implementation-dispatch]
== Onedir and Onefile Dispatch

`luainstaller.bundle(opts)` always builds a dependency plan and manifest
first. Then it dispatches by mode:

[source,text]
----
mode = "onedir"  -> bundler.bundleOnedir(opts)
mode = "onefile" -> onefile.bundleOnefile(opts)
----

Onefile mode stages an ordinary onedir bundle in a temporary directory,
embeds that staged file set into an extractor, and compiles the outer
executable. This is why onefile behavior should be debugged by first
building the corresponding onedir bundle.

[#implementation-errors]
== Error and Logging Strategy

The public error names are part of the user-facing contract. Tests check
representative failures so documentation and CLI output can stay stable.

[cols="1,3",options="header"]
|===
|Error type |Typical source

|`InvalidOptionsError`
|Missing entry, unknown mode, invalid CLI option, or malformed numeric
value.

|`ScriptNotFoundError`
|Entry script or included Lua file does not exist.

|`DynamicRequireError`
|Static analysis found a require call whose name cannot be known
statically.

|`ModuleNotFoundError`
|A requested module cannot be resolved through the active Lua paths.

|`DependencyLimitExceededError`
|Recursive discovery exceeded `--max-deps`.

|`DuplicateModuleError`
|Two files would occupy the same manifest destination.

|`InvalidOutputError`
|The selected output path is unsafe to overwrite.

|`ToolchainError`
|Compiler, Lua prefix, Lua headers, Lua runtime, or profile requirements
are missing.

|`CompilationFailedError`
|The compiler was found but returned a non-zero status.

|`RequireEngineError`
|Runtime discovery could not execute or parse its temporary trace.
|===

Logs are serialized as Lua source under:

[source,text]
----
~/.luainstaller/logs.lua
----

Each entry records timestamp, level, source, action, message, and
details. The newest 1000 entries are kept. Logs help diagnose recent
operations; the manifest remains the authoritative record for a
specific bundle.

[#implementation-maintainer]
== Maintainer Checklist

When changing implementation behavior, update the matching user-facing
contract:

* CLI syntax or output: update `docs/USAGE.adoc`, `luainstaller.1`, and
  `test/cli_split_smoke.lua` or the CLI section of `test/smoke_all.lua`.
* Manifest fields: update this guide, bundle inspection examples, and
  manifest assertions in tests.
* Target profiles: update `docs/PLATFORMS-NATIVE-LIMITS.adoc` and
  remote matrix notes.
* Output safety: update `docs/BUNDLING.adoc`, troubleshooting entries,
  and release safety assertions.
* Error names: update `docs/TROUBLESHOOTING.adoc` and any CLI/API tests
  that assert exact error text.
