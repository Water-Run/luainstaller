# Implementation Research

This note records the current project map, external references, local
experiments, and implementation direction.

## Current Repository Map

- `src/analyzer.lua` owns static `require` extraction, path resolution over
  `package.path` and `package.cpath`, native module detection, cycle detection,
  trace records, and topological sorting.
- `src/manifest.lua` builds the bundle contract used by analyzer, bundler,
  launcher work, diagnostics, and future onefile packaging.
- `src/runtime.lua` contains the reusable pure Lua runtime searcher.
- `src/cgen.lua` builds embedded Lua payloads and generated bootstrap chunks.
- `src/launcher.lua` generates C source that embeds the bootstrap as explicit
  byte arrays.
- `src/launcher/luai_launcher.c` is the shared-Lua C launcher template.
- `src/bundler.lua` implements the `--onedir` filesystem and toolchain layer for
  Linux, macOS, and the Windows MinGW profile.
- `src/init.lua` is the structured public Lua API boundary.
- `src/cli.lua` provides the installed `luai` and `luainstaller` commands plus
  compatibility aliases.
- `luainstaller-1.0.0-1.rockspec` installs both executables and all current Lua
  modules.

## External Reference Map

`luastatic` remains the closest small reference. Its useful idea is generating C
that embeds Lua code, creates a Lua state, installs a searcher, prepares `arg`,
and runs the entry chunk. `luainstaller` reuses that shape but does not simply
wrap `luastatic`.

`luapak` is a broader reference. It proves that LuaRocks-aware packaging and C
module integration can be much more complete, but it is larger than this
project's first Linux milestone.

Evo's standalone executable documentation is useful for the future `--onefile`
extraction model: native libraries must become real files before Lua can load
them through normal native module mechanisms.

References:

- https://github.com/ers35/luastatic
- https://github.com/jirutka/luapak
- https://evo-lua.github.io/docs/how-to-guides/standalone-executables/

## Verified Local Environment

- Lua: 5.4.8.
- LuaRocks: 3.13.0.
- C compiler: local `cc`.
- Lua build flags: `pkg-config --cflags --libs lua`.
- Container clean-run check: a `python:3.11-slim` container without a `lua`
  command loaded the copied `liblua-5.4.so` from `.luai/native/`.

## Implemented Onedir Shape

The first implemented bundle target was Linux `--onedir`, and the same manifest
and launcher pipeline now also covers macOS and Windows profiles:

```sh
luai b --dir test/savinglua/main.lua -o /tmp/savinglua-bundle
luai b --dir test/runtime_bundle/main.lua -o /tmp/win-runtime --target-os windows --lua-prefix /tmp/luainstaller-win-lua
```

The generated directory contains:

```text
app/
    app
    .luai/
        manifest.lua
        native/
            liblua-5.4.so
            <native Lua C modules>
        build/
            launcher.c
```

Linux executables are linked with `$ORIGIN/.luai/native` as a runtime library
search path. The bundler copies the linked Lua shared runtime and detected
native Lua C modules into `.luai/native/`.

macOS links against `liblua.a` from a selected Lua prefix and uses
`@loader_path/.luai/native` for native Lua C modules. Windows is currently built
from Linux with `x86_64-w64-mingw32-gcc`; it emits a `.exe`, copies `lua54.dll`
beside the launcher and into `.luai/native/`, and copies native Lua module DLLs
such as `cjson.dll`, `lsqlite3.dll`, `lfs.dll`, and `socket/core.dll`.

## Main Risks

- Lua C module ABI mismatch must be reported as a compatibility boundary, not
  hidden as a generic load failure.
- External shared library dependencies are not automatically closed. A copied
  Lua C module can still need system libraries such as `libsqlite3.so.0`,
  `libssl.so.3`, or `libcrypto.so.3`.
- Optional dependencies should remain visible in trace output without turning
  intentional capability probes into hard failures.
- Cross-distribution and general cross-building output are not promised. The
  current boundary is same OS family, architecture, C ABI, Lua ABI, and
  compatible system libraries or native DLLs.
- Windows XP and old-system support remain build profiles requiring separate
  toolchains and real smoke runs.

## Next Implementation Questions

- Should `--onefile` store payloads as an appended archive or generated byte
  arrays?
- Should Linux external library handling stay explicit through `--include`, or
  should a diagnostic-only dependency scanner come first?
- Should manifest output gain a stable machine-readable format in addition to
  Lua table serialization?
- Should legacy compatibility commands be removed before a release candidate?
