[#implementation-flow]
== Request flow

Every public packaging request follows the same front half:

[source,text]
----
CLI or Lua API
  -> normalize and validate options
  -> discover dependencies
  -> build compatibility diagnostics
  -> construct manifest
  -> dispatch to onedir or onefile
----

`src/init.lua` owns that orchestration. Submodules return structured errors or
throw internal error tables that are converted at their boundary. CLI rendering
is kept out of the core modules.

[#implementation-api]
== Public API contract

The main module exports:

* `analyze(opts)`;
* `trace(opts)`;
* `compatibility(opts)`;
* `bundle(opts)`;
* `getLogs(opts)`;
* `clearLogs()`.

Success values have `ok = true`. Failures have this shape:

[source,lua]
----
{
    ok = false,
    error = {
        type = "InvalidOptionsError",
        message = "entry is required",
        option = "entry",
    },
}
----

Option validation rejects unknown keys, wrong scalar types, sparse lists, and
unsupported modes before discovery begins. Do not add a CLI option without also
adding it to API normalization and documentation contracts.

[#implementation-modules]
== Module responsibilities

`src/cli.lua`::
Parses the two CLI grammars and renders results. Source-tree preloads allow the
checkout to run without installation.

`src/init.lua`::
Validates the API options, obtains a dependency plan, creates the manifest, and
selects onedir or onefile.

`src/analyzer.lua`::
Lexes literal `require` calls, resolves Lua/native candidates, records trace
decisions, detects cycles, and enforces the dependency limit.

`src/discovery.lua`::
Selects static, runtime, or manual discovery and applies include/exclude rules.
Runtime discovery generates and executes a temporary tracer.

`src/manifest.lua`::
Maps sources to bundle destinations, hashes inputs, and rejects duplicate
destinations.

`src/cgen.lua`, `src/runtime.lua`::
Create and exercise the embedded Lua payload. They contain parallel runtime
logic; changes to searcher installation, source stripping, or `arg` handling
must be made and tested in both places.

`src/launcher.lua`::
Embeds the bootstrap bytes in the platform-aware C launcher template.

`src/bundler.lua`::
Builds onedir output transactionally, compiles the launcher, copies native
files and Lua runtime, and writes the manifest/ownership marker.

`src/onefile.lua`::
Stages onedir, generates the extractor, enforces extraction-path safety, and
starts the cached inner launcher.

`src/path.lua`, `src/process.lua`, `src/result.lua`::
Shared normalization, shell execution/quoting, and error constructors.

`src/platform.lua`, `src/compat.lua`::
Host detection, target profiles, and user-facing compatibility diagnostics.

`src/logger.lua`::
Best-effort persistent failure logging below the user home directory.

[#implementation-static]
== Static discovery

The lexer is deliberately narrower than a full Lua parser. It tracks comments,
quoted strings, and long-bracket strings so that text such as
`-- require("x")` is ignored. It recognizes:

* `require("name")` and `require "name"`;
* single quotes and long-bracket literals;
* `pcall(require, "name")` as optional;
* `local require = require` followed by normal literal calls.

Quoted literal contents are decoded with Lua's text parser in an empty
environment. Concatenation and computed arguments produce
`DynamicRequireError`.

`ModuleResolver` searches entry-relative templates, then active `package.path`
and `package.cpath` templates. Trace records retain the requesting file,
requested name, candidates, selected path/type, and resolution reason.

The recursive analyzer marks a source visited only after its children complete.
This makes cycles visible and produces dependency-first ordering for the
manifest.

[#implementation-runtime-discovery]
== Runtime discovery

Runtime mode creates a private `0700` work directory and writes a generated Lua
tracer. The tracer patches global `require`, extends search paths relative to
the entry, then executes the entry with the supplied run arguments.

A require record is provisional until the real `require` succeeds. If loading
fails and the application catches the error with `pcall`, that provisional
record is removed. Records are flushed before `os.exit`.

After execution, recorded names are resolved through the same `ModuleResolver`
used by static mode. This produces a common dependency/trace shape for later
stages.

The mode cannot observe unexecuted branches or a `require` reference captured
before the tracer patch. Manual includes are the escape hatch for both cases.

[#implementation-manifest]
== Manifest mapping

Pure-Lua sources preserve paths relative to the entry directory. Files outside
that tree are placed below a sanitized `external/` path. Native modules start
with basename destinations; trace information may add a dotted-module
destination.

Manifest content hashes and generated-output hashes use the shared
`manifest.fnv1a32` implementation. The algorithm name is recorded explicitly.
Do not change it without a marker/manifest version decision.

`manifest.compatibility` is a packaging requirement list, not a snapshot of the
build host. Structured host/target values are returned by `trace` and
`compatibility`.

[#implementation-bootstrap]
== Generated launcher and bootstrap

`launcher.generateSource` combines three pieces:

. a byte array containing the generated Lua bootstrap;
. a C launcher template;
. platform code that resolves the current executable path.

The C launcher sets `arg`, installs a traceback handler, and calls the bootstrap
with `lua_pcall`. The bootstrap prepends native search paths, installs the
embedded module searcher at index 2, and runs the entry. Both the entry and
modules are loaded in text mode.

The embedded runtime restores `_G.arg` and removes its package searcher after an
in-process run. The native process exits immediately afterward, but retaining
the cleanup makes `src/runtime.lua` safe for tests and API use.

[#implementation-onedir]
== Onedir transaction and ownership

The final output is validated before staging starts. A private sibling directory
is then created with an unpredictable name and `0700` mode. All potentially
failing generation, compilation, copy, and hash operations happen there.

At commit time:

. an existing generated output is renamed to a unique backup;
. the complete staging directory is renamed to the final path;
. the backup is removed;
. if the second rename fails, the backup is restored.

The marker records the final path even though it is written in staging. This is
required for later ownership validation.

The overwrite checks intentionally distrust the generated comment alone. They
also verify the marker path, top-level allowlist, and file hashes.

[#implementation-onefile]
== Onefile security boundary

The extractor accepts only safe relative embedded paths. On POSIX it canonicalizes
the configured temporary root, creates a UID-specific private cache, and checks
owner/mode with `lstat`. On Windows it rejects reparse-point directories.

Every existing directory component must be a real directory. A symlink at the
final file is removed; a symlink in a parent path aborts extraction. Cached files
are accepted only when size and hash match.

Windows child command lines use the CRT quoting rule: ordinary backslashes are
preserved, backslashes before a quote are doubled appropriately, and trailing
backslashes are doubled before the closing quote.

[#implementation-errors]
== Error boundaries

Use a specific error type when the caller can act on it:

* `ScriptNotFoundError` — entry/include/source missing;
* `DynamicRequireError` — static discovery cannot prove the name;
* `ModuleNotFoundError` — no candidate resolves;
* `DuplicateModuleError` — destination or module-name collision;
* `InvalidOutputError` — output ownership/safety check failed;
* `ToolchainError` — required compiler, headers, prefix, or `pkg-config` missing;
* `CompilationFailedError` — compiler ran and rejected the source/link;
* `FilesystemError` — create, copy, hash, rename, or extraction preparation
  failed;
* `DiscoveryError` — runtime tracer execution or output failed.

Shell commands must quote every path and reject compiler flag metacharacters.
Never place an unquoted value derived from `arg`, environment variables, or a
manifest in a command string.

[#implementation-change-checklist]
== Before merging an implementation change

. Add a test that fails for the old behavior.
. Run syntax checks and all three local test entry points.
. Inspect generated C with warning flags when launcher/extractor code changes.
. Run every affected remote target; generated C changes affect all targets.
. Update the usage or format documentation when a public contract changes.
. Confirm a failed rebuild cannot damage an existing generated output.
. Confirm clean-environment execution does not fall back to host Lua modules.

The exact commands are in xref:TESTING.adoc[Testing].
