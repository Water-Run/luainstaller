[#usage-session]
== A Normal Packaging Session

Start from a Lua program that already runs with the Lua interpreter you
intend to build with. Packaging is much easier to debug when the source
program is known to work first.

[source,sh]
----
lua app/main.lua --help
----

Then run the packaging workflow in this order:

[source,text]
----
run with lua
  -> analyze dependencies
  -> trace suspicious dependency decisions
  -> build --dir
  -> inspect manifest and native files
  -> run the directory bundle
  -> build --file only after --dir works
----

The directory bundle is the debugging format. It leaves the generated C
source, manifest, copied native modules, and runtime files on disk. The
onefile bundle is a distribution format layered on top of that same
layout.

[#usage-cli-surfaces]
== Compact and Full-Word CLIs

`luai` and `luainstaller` are deliberately different command surfaces.
They call the same Lua API, but their syntax is not interchangeable.

[cols="1,2,2,3",options="header"]
|===
|Action |`luai` |`luainstaller` |Use it when

|Help
|`luai -h`
|`luainstaller help`
|Checking the accepted syntax for that command surface.

|Version
|`luai -v`
|`luainstaller version`
|Verifying an install or recording issue data.

|Analyze
|`luai -a <entry.lua>`
|`luainstaller analyze <entry.lua>`
|Seeing what Lua source and native modules will be considered.

|Trace
|`luai -t <entry.lua>`
|`luainstaller trace <entry.lua>`
|Explaining why a module resolved, was optional, or failed.

|Build
|`luai -b <entry.lua>`
|`luainstaller build <entry.lua>`
|Creating an onedir or onefile bundle.

|Logs
|Not exposed
|`luainstaller logs`
|Reading persistent operation logs.
|===

[IMPORTANT]
====
`luai -b app.lua` and `luainstaller build app.lua` are valid.
`luai build app.lua` and `luainstaller -b app.lua` are invalid. This is
intentional: scripts get a small compact surface, while the full-word
CLI remains explicit for humans.
====

[#usage-first-bundle]
== First Bundle

Use the pure-Lua runtime sample for a quick local check:

[source,sh]
----
luai -a test/runtime_bundle/main.lua --max-deps 120
luai -t test/runtime_bundle/main.lua --max-deps 120
luai -b --dir test/runtime_bundle/main.lua -o build/runtime-demo --max-deps 120
build/runtime-demo/runtime-demo Ada
----

Expected analysis output is small:

[source,text]
----
ok
entry: test/runtime_bundle/main.lua
scripts: 1
libraries: 0
----

Expected trace output includes the compatibility boundary:

[source,text]
----
trace
entry: test/runtime_bundle/main.lua
1: lua resolved greeter -> ...
compatibility: same OS, same architecture, same ABI, same Lua ABI
note: does not claim universal cross-platform output
----

After the directory bundle runs, build the onefile wrapper:

[source,sh]
----
luai -b --file test/runtime_bundle/main.lua -o build/runtime-onefile --max-deps 120
build/runtime-onefile Ada
----

[#usage-native-bundle]
== Native Module Bundle

Native Lua C modules are the first place where packaging stops being
pure file copying. Build the directory form first and inspect it before
trying a onefile:

[source,sh]
----
luainstaller trace test/student_management_system/main.lua --max-deps 250
luainstaller build --dir test/student_management_system/main.lua \
  -o build/student-manager \
  --max-deps 250

find build/student-manager/.luai/native -maxdepth 3 -type f | sort
build/student-manager/student-manager --data /tmp/students.json seed
----

If a native module appears in `.luai/native/` but the executable fails
at startup, the missing piece is often an external library used by that
module, not the Lua module file itself. Use `ldd`, `otool -L`, or a
Windows DLL dependency tool as appropriate.

[#usage-options]
== Option Reference

[cols="1,1,3",options="header"]
|===
|Option |Applies to |Behavior

|`--dir`, `--onedir`
|Build
|Create a directory bundle. This is the default and the recommended
first build format.

|`--file`, `--onefile`
|Build
|Create a self-extracting onefile wrapper around a staged directory
bundle.

|`-o PATH`, `--out PATH`
|Build
|Choose the output directory for `--dir` or executable path for
`--file`.

|`--include PATH`
|Analyze, trace, build
|Add an extra Lua source file. Repeat it for multiple files. Manual
includes are still subject to duplicate-destination checks.

|`--exclude PATH`
|Analyze, trace, build
|Exclude a resolved dependency by exact path, basename, or suffix.
Excludes win over automatic discovery and manual includes.

|`--target-os OS`
|Trace, build
|Select `linux`, `macos`, or `windows`. The selected profile determines
compiler behavior and runtime layout.

|`--lua-prefix PATH`
|Trace, build
|Point macOS and Windows profiles at a Lua prefix containing headers and
the required Lua runtime files.

|`-d MODE`, `--discovery-mode MODE`
|Analyze, trace, build
|Select `static`, `manual`, or `runtime` dependency discovery mode.

|`--no-depscan`
|Analyze, trace, build
|Disable automatic scanning. This is equivalent to the manual discovery
mode.

|`--max-deps N`
|Analyze, trace, build
|Set the dependency limit. The default is `36`; samples with native
packages often use a larger value.

|`--verbose`
|Analyze, build
|Print additional trace or manifest-size detail where the command
supports it.

|`--`
|Runtime discovery
|Stop option parsing and pass the remaining values to the entry program
during a build-time runtime trace.
|===

[#usage-discovery]
== Choosing a Discovery Mode

[cols="1,2,3",options="header"]
|===
|Mode |Example |Choose it when

|`static`
|`luai -b app/main.lua -o build/app`
|Your `require` calls use literal module names that can be read from the
source code.

|`runtime`
|`luai -b app/main.lua -d runtime -o build/app -- demo-profile`
|Module names are assembled dynamically, but one build-time run can
exercise the paths you need in the bundle.

|`manual`
|`luai -b app/main.lua --no-depscan --include app/plugin.lua -o build/app`
|You want complete control, or static/runtime discovery would include
too much or too little.
|===

[WARNING]
====
`--discovery-mode runtime` executes the entry program once during dependency
discovery. It only records `require` calls reached by that run. If your
program loads different modules for different subcommands, run a
representative command or use manual includes for the missing files.
====

[#usage-inspection]
== Inspecting a Bundle

The manifest is Lua data. Load it with the same Lua version family used
for development:

[source,sh]
----
lua -e 'local m = dofile("build/student-manager/.luai/manifest.lua"); print(m.output.mode); print(#m.modules.lua); print(#m.modules.native)'
----

Trace records are useful when a module appears under an unexpected path:

[source,sh]
----
lua - <<'LUA'
local m = dofile("build/student-manager/.luai/manifest.lua")
for _, item in ipairs(m.trace or {}) do
  print(item.classification, item.requested, item.reason, item.selected_path or "")
end
LUA
----

Run a POSIX directory bundle with a sparse environment to catch hidden
dependencies on your shell configuration:

[source,sh]
----
env -i PATH=/usr/bin:/bin build/runtime-demo/runtime-demo clean
----

[#usage-logs]
== Logs

Operational logs are stored under `~/.luainstaller/logs.lua` and read
through the full-word CLI:

[source,sh]
----
luainstaller logs
luainstaller logs --limit 20
luainstaller logs --level error
----

Use logs to recover the error type, command, and details after a failed
operation. Do not treat logs as a build manifest; the generated bundle
manifest is the authoritative record of what was packaged.

[#usage-api]
== Lua API

Load the public module:

[source,lua]
----
local luainstaller = require("luainstaller")
----

Public calls return structured result tables. Normal user errors are
reported as `{ ok = false, error = ... }`; callers do not need `pcall`
for expected packaging failures.

[source,lua]
----
local result = luainstaller.analyze({
  entry = "test/student_management_system/main.lua",
  max_deps = 250,
})

if result.ok then
  print(#result.dependencies.scripts, #result.dependencies.libraries)
else
  io.stderr:write(result.error.type .. ": " .. result.error.message .. "\n")
end
----

[cols="1,3",options="header"]
|===
|Function |Result

|`luainstaller.analyze(opts)`
|Dependency lists and trace records for the selected discovery mode.

|`luainstaller.trace(opts)`
|Dependency trace plus compatibility diagnostics for the selected
profile.

|`luainstaller.compatibility(opts)`
|Host/target profile, Lua ABI, compatibility notes, and native-module
warnings.

|`luainstaller.bundle(opts)`
|Bundle mode, executable path, generated manifest, or a structured
error.

|`luainstaller.getLogs(opts)`
|Filtered persistent log entries.

|`luainstaller.clearLogs()`
|Clears persistent log entries.
|===

Common error types include `InvalidOptionsError`,
`ScriptNotFoundError`, `DynamicRequireError`, `ModuleNotFoundError`,
`DependencyLimitExceededError`, `DuplicateModuleError`,
`InvalidOutputError`, `ToolchainError`, `CompilationFailedError`,
`LuaRuntimeNotFoundError`, `DiscoveryError`, and
`LauncherGenerationError`.
