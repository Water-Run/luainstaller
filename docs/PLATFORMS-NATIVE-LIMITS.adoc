[#platforms-contract]
== Native-host contract

`luainstaller` removes the deployed program's dependency on a separately
installed `lua` command. It does not emulate another operating system or
translate native ABIs. A generated bundle must run on a target compatible with
the build host in all of these dimensions:

. operating-system family and loader conventions;
. CPU architecture;
. system and compiler ABI;
. Lua major/minor ABI;
. native-module dependencies.

Build on the target OS and architecture. `--target-os`, when supplied, is an
assertion about the detected native host; asking for another OS or architecture
returns `UnsupportedPlatformError`. There is no cross-build mode.

The supported interpreter range is official Lua `>= 5.1` and `< 6.0`. LuaJIT
is rejected. The analysis interpreter, runtime tracer, header set, linked Lua
runtime, and copied Lua C modules must share one major/minor ABI. Patch versions
may differ only where the upstream ABI remains compatible; test the exact
runtime and modules that will be shipped.

Launcher C is generated for the active ABI. For example, a build running under
Lua 5.3 uses a `LUA_VERSION_NUM == 503` compile-time guard, `lua5.3` module
paths, and an ABI-specific runtime name. Before output is published,
luainstaller compiles and executes a linked-runtime probe and requires the same
`Lua 5.x` result as the build interpreter. The launcher repeats the runtime
check when it starts.

[#platforms-toolchains]
== Native toolchain discovery

Toolchain selection is capability-based on the current host. The tested native
paths are:

[cols="1,2,3",options="header"]
|===
|Host family |Lua runtime strategy |Required build capability

|Linux
|Shared Lua runtime copied into `.luai/native`
|Native C compiler plus a matching `pkg-config` Lua record and shared library

|macOS
|Lua static library linked into the launcher
|Native Apple compiler plus a matching prefix containing headers and
`lib/liblua.a`

|Windows
|ABI-specific Lua DLL copied beside the executable and into `.luai/native`
|Native MSVC toolchain, Windows SDK, headers, import library, and matching DLL
|===

These rows explain the backends exercised by this repository. They do not
restrict luainstaller to particular distribution names, computer models, or OS
versions. A host that supplies the required native capability still needs its
own release test.

Use `--lua-prefix PATH` or `LUAI_LUA_PREFIX` when automatic discovery cannot
identify the intended Lua installation. The prefix must belong to the current
host and the interpreter's major/minor ABI. It is not a cross-compilation SDK.
Once selected explicitly, that prefix is authoritative and discovery does not
silently substitute another installed Lua.

On Linux, `pkg-config --modversion` and compiler/linker flags are checked
against the active Lua ABI. A static-only Lua installation is insufficient for
the shared-runtime profile. For a custom installation, expose its matching
`.pc` file:

[source,sh]
----
PKG_CONFIG_PATH=/opt/lua/lib/pkgconfig \
  luainstaller build --dir app/main.lua -o build/app
----

On macOS, a typical prefix contains:

[source,text]
----
/opt/lua/
├── include/lua.h
└── lib/liblua.a
----

The launcher and onefile extractor retain the linker's content-derived
`LC_UUID`. Lua C modules remain Mach-O bundles with their own architecture and
deployment-target requirements. Inspect them with `file` and `otool -L`.

On Windows, use the ABI-specific runtime and import library selected by the
native toolchain, such as `lua51.dll`/`lua51.lib` or
`lua55.dll`/`lua55.lib`. The build and linked-runtime probe run directly on
Windows; Wine and MinGW are neither required nor used by the release gate.
Windows paths are checked without following reparse points. Generated runtime
code currently uses narrow Win32 path/process APIs, so release tests must cover
every non-ASCII path and argument repertoire the application promises.

[#platforms-native]
== Native Lua C modules

Discovery copies the module selected from `package.cpath`. It does not:

* rebuild the module for another OS, architecture, or Lua ABI;
* rewrite RPATH or install names;
* recursively copy OpenSSL, database, C++ runtime, or vendor libraries;
* verify license or redistribution terms.

Use the source program's actual module paths during analysis. In these examples,
replace `5.x` with the ABI being built:

[source,sh]
----
LUA_PATH='/srv/app/share/lua/5.x/?.lua;;' \
LUA_CPATH='/srv/app/lib/lua/5.x/?.so;;' \
luai -t app/main.lua --max-deps 250
----

Inspect copied native files and their dependent libraries before release. Use
`ldd` on Linux, `otool -L` on macOS, and a native PE dependency viewer on
Windows. A successful launcher compile proves the Lua runtime link, not closure
of every application module dependency.

[#platforms-limits]
== Product boundaries

The current release does not provide:

* cross-compilation or universal cross-platform executables;
* automatic recursive closure of native system-library dependencies;
* bytecode-only packaging, encryption, or source obfuscation;
* direct loading of native modules from onefile without extraction;
* universal macOS binaries, code signing, or notarization;
* MSI, PKG, DEB, RPM, or other installer generation.

Onefile is an executable packaging form, similar in intent to PyInstaller's
single-file mode: it extracts a verified native onedir payload before starting
the inner launcher. It is not an operating-system installer.
