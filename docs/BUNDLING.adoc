[#bundling-onedir]
== Directory layout

Linux / macOS:

[source,text]
----
app/
├── app
├── THIRD_PARTY_NOTICES.md
└── .luai/
    ├── build/
    │   ├── generate-onefile-payload.lua
    │   ├── launcher.c
    │   └── RELINKING.adoc
    ├── generated-output.txt
    ├── licenses/
    │   ├── GPL-3.0-or-later.txt
    │   ├── LGPL-3.0-or-later.txt
    │   └── Lua-MIT.txt
    ├── manifest.lua
    └── native/
        ├── liblua-5.x.so    # shared-Lua profiles; name follows ABI
        └── module.so
----

Windows places the Lua DLL beside the executable and a second copy under
`.luai/native/` for dependent modules.

The entry and pure-Lua modules are compiled into the C launcher and listed in
the manifest; they are not left as loose `.lua` files in the tree.

[#bundling-build]
== Build placement

Generation, copy, compile, and manifest write happen in a private staging
directory, then rename into the final path. When replacing a generated bundle,
the old tree is backed up and revalidated; on failure the previous bundle is
restored when possible.

The output lock only coordinates `luainstaller` itself. On POSIX, another
process holding a writable descriptor opened before rename may still mutate
that inode after verification. Use a new output path when that cannot be
ruled out.

Each lock binds a cryptographic token, owner PID, output hash, and exact
staging name. Normal failures and SIGINT-visible interruptions finalize owned
state; the CLI reports such an interruption with status `130`. After an
uncatchable crash or SIGKILL, the next build reclaims only an
unchanged lock whose owner PID is proven dead; live, malformed, or replaced
locks are left untouched.

[#bundling-launcher]
== Startup

. Create a Lua state and open standard libraries
. Set `arg`
. Load the embedded bootstrap as text-only Lua
. Run the entry with a traceback handler

The process image path comes from the OS (not the spelling of `argv[0]`), so
launch via `PATH` or a symlink still locates `.luai/native`.

Application code sees the launcher's invoked executable spelling in
`arg[0]`; it is never replaced with the build-time entry-script path. The
OS-resolved process image used for native lookup is private bootstrap state and
is cleared before the entry runs.

[#bundling-lua]
== Pure-Lua lookup order

. `package.preload`
. Modules embedded in the bundle
. Filesystem Lua searchers
. Native searchers

Packaged modules take precedence without a hermetic sandbox: undeclared names
may fall through to the host. Embedded code uses `load(..., "t")` (no bytecode
payload). Name collisions → `DuplicateModuleError`.

[#bundling-native]
== Native modules

Copied by basename and, when known, dotted path—for example `socket.core`:

[source,text]
----
.luai/native/core.so
.luai/native/socket/core.so
----

`package.cpath` is prepended with `.luai/native` patterns for `?.so` /
`?.dylib` / `?.dll` and `?/init.*`. RPATH covers the copied Lua runtime;
further system libraries of a C module are not auto-packaged. Before publish, a
native linked probe requires `_VERSION` to match the build interpreter.

[#bundling-manifest]
== Manifest and ownership

`.luai/manifest.lua` records entry, hashes, mode, Lua ABI, platform, module
groups, include/exclude, and trace data. Format `version = 2` with
`hash_algorithm = "sha256"`. Artifact manifests identify sources with stable
`source_id` values: paths below the entry directory are relative, while an
external source uses `external/<sha256>/<basename>`. Build-host absolute
paths, the output path, and manual include paths remain only in the in-memory
build result and diagnostics; they are not serialized into the bundle.
Sources are re-read before embed/copy; mismatch → `SourceChangedError`.

With identical inputs and toolchain, builds made under different checkout
roots produce byte-identical onefile artifacts. Generated bootstrap chunk
names and manifest trace references use the same logical source IDs, so
checkout directories and build usernames are not embedded as metadata.

[#bundling-licenses]
== Licenses and corresponding source

Every onedir and onefile payload contains the Lua MIT license, luainstaller's
LGPL-3.0-or-later text, the GPLv3 text incorporated by the LGPL, and
`THIRD_PARTY_NOTICES.md`. The manifest records each material's destination and
SHA-256; canonical material is installed with the Lua module so building from
another working directory does not depend on repository-relative files.

`.luai/build/launcher.c` is the exact generated inner-launcher source and
contains bundled Lua sources as text. Onefile payloads additionally contain
`.luai/build/extractor.c`, the exact outer extractor translation unit, with a
documented generated `payload.inc` interface, `payload-files.lua`, and a
standalone `generate-onefile-payload.lua` re-generator. Preserve
`.luai/build/RELINKING.adoc` and the complete application/module sources when
redistributing. Extra dependencies may require additional notices or source.

`.luai/generated-output.txt` is the v2 ownership marker
(`luainstaller-generated-output-v2`): final directory, every generated
directory, and every generated regular file with SHA-256. Replacement is
allowed only when the **recursive** inventory still describes the
**exact file set** (hashes included). Tree edits, symlinks, or record mismatch refuse
automatic overwrite. Markers from `luainstaller-generated-output-v1` are never
auto-upgraded; migration is a **one-time removal** after backup, or a new
output path.

[#bundling-onefile]
== Single-file

Build a full directory bundle first, add the extractor corresponding source,
then embed runtime files into the extractor (path, size, executable bit,
bytes). The onedir ownership marker is omitted; `.luai/build/` is retained for
license and relinking purposes. An existing output path is not overwritten.

POSIX cache: `luainstaller-onefile-<effective-uid>` under the canonical
temporary root; permission and sticky-ancestor rules are in
xref:TROUBLESHOOTING.adoc[Troubleshooting]. Windows uses the user temporary
directory with reparse/DACL constraints.

After extraction, POSIX replaces the outer process with the inner launcher,
preserving PID, signal, and exit semantics. Windows starts the inner launcher
suspended, assigns it to a kill-on-close Job Object, and resumes it only after
containment succeeds; forcefully terminating the outer process therefore also
terminates the inner process tree.

Cache hits require full-byte equality. After confirming no process uses the
cache:

[source,bash]
----
rm -rf "${TMPDIR:-/tmp}/luainstaller-onefile-$(id -u)"
----

If single-file fails, verify the matching directory bundle for the same entry
first.
