after the equivalent onedir bundle has been tested.

[#troubleshooting-first]
== First five commands

[source,sh]
----
lua app/main.lua --help
luai -a app/main.lua --max-deps 120
luai -t app/main.lua --max-deps 120 --verbose
luai -b --dir app/main.lua -o build/app --max-deps 120
env -i PATH=/usr/bin:/bin build/app/app
----

Record the first command that fails. Do not move to onefile until all five pass.

[#troubleshooting-cli]
== The command is rejected immediately

Symptoms::
`unknown luai command`, `unknown luainstaller command`, or `unknown option`.

Checks::

* `luai` uses `-a`, `-t`, and `-b`.
* `luainstaller` uses `analyze`, `trace`, and `build`.
* Options that take values must be followed by a value.
* Entry-script arguments for runtime discovery follow `--`.

[source,sh]
----
luai -b app/main.lua -- --profile staging
luainstaller build app/main.lua -- --profile staging
----

[#troubleshooting-errors]
== Structured error types

`ScriptNotFoundError`::
The entry or a manual include cannot be opened. Check the working directory,
case, and whether the path names a regular file.

`DynamicRequireError`::
Static discovery found a computed name or unsupported alias. Use runtime mode
for a safe executable path, or manual mode with reviewed includes.

`ModuleNotFoundError`::
No Lua or native candidate exists in the active search templates. Compare the
trace candidates with the source program's `package.path` and `package.cpath`.

`DependencyLimitExceededError`::
The number of discovered Lua files exceeded `--max-deps`. Inspect the trace for
an unexpectedly broad module tree before raising the limit.

`DuplicateModuleError`::
Two sources map to one module name or bundle destination. Preserve directory
structure for manual includes and remove ambiguous copies from search paths.

`InvalidOptionsError`::
An API value has the wrong type, a list is sparse, a mode is unknown, or a key
is unsupported. CLI spelling errors are usually caught before the API call.

`InvalidOutputError`::
The output already contains user data, has been modified since generation, is a
symlink, or does not match its ownership marker. Inspect it; do not bypass the
check blindly.

`ToolchainError`::
A required compiler, Lua prefix, shared runtime, or `pkg-config` record is
missing.

`CompilationFailedError`::
The compiler ran but failed. Read the captured command and output from the first
diagnostic line, not only the final linker status.

`LuaRuntimeNotFoundError`::
The Linux launcher linked without a discoverable shared Lua library. Confirm
that `pkg-config --libs lua` selects shared Lua and that `ldd` can resolve it.

`DiscoveryError`::
The runtime tracer entry failed, exited nonzero, could not create its private
work directory, or did not produce parseable output.

`FilesystemError`::
A write, copy, hash, staging rename, cache validation, or cleanup failed. Check
the `path`, `backup_path`, and `restore_error` fields when present.

[#troubleshooting-static]
== Analysis does not match runtime

Print the interpreter's real search paths:

[source,sh]
----
lua -e 'print(package.path); print(package.cpath)'
----

Run analysis with those same environment variables. A common mismatch is using
one Lua installation to run the source and a different installation to run
`luai`.

For static discovery, check whether the source uses:

* a computed module name;
* a different-name alias of `require`;
* module path setup performed before the entry runs;
* escaped characters in a module name;
* optional loads other than `pcall(require, "name")`.

Use `-t --verbose` to see the requested name, candidates, and selected path.

[#troubleshooting-runtime]
== Runtime discovery fails or misses a module

Run the entry manually with the exact discovery arguments first:

[source,sh]
----
lua app/main.lua profile_a input.json
luai -a app/main.lua -d runtime -- profile_a input.json
----

Runtime mode observes only the branch taken by this run. Repeat with other
profiles or add manual includes for modules loaded on later branches.

A missing module caught by the application's `pcall` is not treated as a
dependency. A different runtime error inside a module still fails tracing.

If startup changes data, sends requests, or launches services, do not run it as
a build-time tracer. Use static/manual discovery or add a harmless discovery
mode to the application.

[#troubleshooting-compile]
== Compiler or linker failure

=== Linux

[source,sh]
----
pkg-config --cflags --libs lua
cc --version
lua -v
----

If `pkg-config` cannot find Lua, install the development package or point
`PKG_CONFIG_PATH` at a matching `lua.pc`. If the launcher compiles but
`LuaRuntimeNotFoundError` follows, the selected Lua may be static-only.

=== macOS

[source,sh]
----
test -f "$LUA_PREFIX/include/lua.h"
test -f "$LUA_PREFIX/lib/liblua.a"
file "$LUA_PREFIX/lib/liblua.a"
----

The prefix and native Lua modules must contain the target architecture. An
x86_64 archive does not satisfy an ARM64 build.

=== Windows profile

[source,sh]
----
x86_64-w64-mingw32-gcc --version
find "$LUA_PREFIX" -maxdepth 2 -iname 'lua*.dll' -print
----

The current profile requires a Linux host, MinGW x86_64 compiler, headers, and
a compatible Lua DLL.

[#troubleshooting-native]
== Bundle starts, then a native module fails

Typical messages include `module not found`, `undefined symbol`, `image not
found`, `wrong ELF class`, or `not a valid Win32 application`.

. Confirm the module is present in `.luai/native`.
. Inspect the manifest trace for its requested name and selected source.
. Inspect architecture and external libraries.
. Run with Lua environment variables cleared.
. Test on the actual target OS.

[source,sh]
----
file build/app/.luai/native/*
ldd build/app/.luai/native/module.so
env -i PATH=/usr/bin:/bin build/app/app
----

On macOS use `otool -L`; on Windows use a PE dependency viewer or
`dumpbin /dependents`.

If direct execution works only without the clean environment, the program is
falling back to host modules rather than using the bundle.

[#troubleshooting-path]
== Direct path works but PATH/symlink launch fails

This should be covered by the current launcher, which resolves the process
image path through the operating system. If it regresses, collect:

[source,sh]
----
readlink -f "$(command -v app)"
find /real/bundle/.luai/native -maxdepth 3 -type f -print
env -i PATH=/directory/with/link:/usr/bin:/bin app
----

Confirm that the executable was produced by the current source tree; older
launchers derived native paths directly from `argv[0]`.

[#troubleshooting-output]
== Rebuild refuses an existing directory

Read the marker and list top-level entries:

[source,sh]
----
find build/app -mindepth 1 -maxdepth 2 -print | sort
sed -n '1,160p' build/app/.luai/generated-output.txt
----

Common causes are a user file added beside the executable, an edited executable,
a moved output directory, or an incomplete bundle made by an older release.

Use a new output path when uncertain. A current failed rebuild should leave the
previous generated directory runnable because compilation occurs in staging.

Each onedir build holds a sibling lock named `.<output-name>.luai-lock`. If the
builder is killed or the machine loses power, the hidden directory can remain.
Before removing it, confirm that no `luainstaller`, `luai`, compiler, or matching
build process is still running and inspect its timestamp:

[source,sh]
----
ls -ld build/.app.luai-lock
ps aux | grep -E '[l]uainstaller|[l]uai|[c]c|[g]cc|[c]lang'
rm -rf build/.app.luai-lock
----

Never remove a lock merely because a build is slow; native dependency builds
can legitimately run for several minutes.

[#troubleshooting-onefile]
== Onefile extraction fails

First verify onedir. Then inspect the temporary root:

[source,sh]
----
printf 'TMPDIR=%s\n' "${TMPDIR:-/tmp}"
ls -ld "${TMPDIR:-/tmp}"
ls -ld "${TMPDIR:-/tmp}/luainstaller-onefile-$(id -u)" 2>/dev/null
----

On POSIX the cache must be a real directory owned by the effective user with no
group/other permissions. Parent-directory symlinks inside the cache are
rejected. The temporary root itself is canonicalized, so the standard macOS
`/tmp -> /private/tmp` link is supported.

Remove a cache only after confirming that no bundle process is using it:

[source,sh]
----
rm -rf "${TMPDIR:-/tmp}/luainstaller-onefile-$(id -u)"
----

On Windows, cache directories must be owned by the current user's SID and must
accept the protected user-and-`SYSTEM` DACL. Check `%TEMP%`, reparse points,
path length, antivirus quarantine, domain security policy, and permission to
create a child process.

[#troubleshooting-logs]
== Logs and issue data

[source,sh]
----
luainstaller version
luainstaller logs --level error --limit 20
lua -v
uname -a
----

For a useful report include:

* exact command and exit status;
* complete structured error, compiler output, or traceback;
* host and target OS/architecture;
* Lua version and `pkg-config --cflags --libs lua` where relevant;
* discovery mode and redacted trace;
* onedir manifest and marker;
* `file` plus native dependency inspection;
* whether clean-environment onedir and onefile differ.

Remove credentials, tokens, home-directory details, and proprietary source
before sharing logs or manifests.
