[#platforms-contract]
== What “portable” means here

`luainstaller` removes the target machine's dependency on a separate `lua`
command. It does not make native code universal.

A target must match:

. operating-system family;
. CPU architecture;
. system ABI and loader conventions;
. Lua major/minor ABI;
. native module compiler/runtime assumptions.

A Linux x86_64 bundle is not a Windows or Linux ARM64 bundle. A Lua 5.4 native
module is not loadable by a Lua 5.3 runtime merely because its filename is the
same.

The package metadata allows Lua >= 5.4 and < 5.5, but every concrete toolchain
component must identify as exactly Lua 5.4: the analysis interpreter, runtime
tracer, headers, `pkg-config` record, linked runtime, and native modules all
share that major/minor ABI. LuaJIT is not a supported substitute.

[#platforms-profiles]
== Build profiles

[cols="1,2,3",options="header"]
|===
|Target |Launcher/runtime |Build requirement

|Linux
|Shared Lua launcher; linked Lua shared object copied into `.luai/native`
|Linux host, C compiler, `pkg-config --cflags --libs lua`, shared Lua library

|macOS
|Lua static library linked into launcher
|macOS host and `--lua-prefix` containing `include/lua.h` and `lib/liblua.a`

|Windows x86_64
|MinGW launcher plus Lua DLL beside executable and under `.luai/native`
|Linux host, x86_64 MinGW compiler, Wine, and Windows Lua prefix
|===

`--target-os` selects one of these profiles. It does not cross-compile Lua C
modules found in the source environment; those modules must already match the
target.

All generated launcher sources contain a `LUA_VERSION_NUM == 504` compile-time
guard. Supplying older or newer headers stops at compilation even if a prefix
has the expected filenames. The bundler also executes a program linked to the
selected runtime and requires its `_VERSION` to be exactly `Lua 5.4`; Windows
uses Wine for this pre-publication probe. The launcher repeats that runtime
check on every execution.

[#platforms-linux]
== Linux

The Linux compiler command is derived from `pkg-config` and adds:

[source,text]
----
-Wl,-rpath,'$ORIGIN/.luai/native'
----

After compilation, `ldd` is used to locate the linked `liblua`. A static-only
Lua installation is insufficient for this profile because there is no shared
runtime to copy.

Before compilation, `pkg-config --modversion lua` must parse as 5.4. Compiler
flags from a differently named or mismatched Lua package are rejected rather
than trusted based on `-llua` alone.

For a custom Lua build, expose a temporary or permanent `lua.pc`:

[source,sh]
----
PKG_CONFIG_PATH=/opt/lua-5.4/lib/pkgconfig \
  luainstaller build --dir app/main.lua -o build/app
----

Test on the oldest glibc and distribution baseline you intend to support.
Building on a newer system can introduce symbol versions absent from older
targets.

[#platforms-macos]
== macOS

The macOS profile requires a static Lua prefix:

[source,text]
----
/opt/lua-5.4/
├── include/lua.h
└── lib/liblua.a
----

The launcher and onefile extractor retain the linker's content-derived
`LC_UUID`. Current macOS releases require that load command at execution time;
using the linker's deterministic default also preserves repeated-build byte
identity when the inputs and output basename are unchanged.

[source,sh]
----
luainstaller build --dir app/main.lua \
  --target-os macos \
  --lua-prefix /opt/lua-5.4 \
  -o build/app
----

Lua is static in the launcher, but Lua C modules remain Mach-O bundles with
their own architecture and deployment-target requirements. Inspect them with
`file` and `otool -L`.

The project does not create universal (`arm64` + `x86_64`) binaries, sign code,
or notarize output.

[#platforms-windows]
== Windows

The Windows profile currently targets x86_64 from Linux using MinGW. The prefix
must provide headers and one of these ABI-specific DLL locations:

* `bin/lua54.dll`;
* `lua54.dll`.

A generic `lua.dll` is not accepted because its filename does not prove the
required 5.4 import ABI.

[source,sh]
----
luainstaller build --dir app/main.lua \
  --target-os windows \
  --lua-prefix /opt/windows-lua54 \
  -o build/app
----

Set `LUAI_WINDOWS_CC` to override `x86_64-w64-mingw32-gcc`. The Lua DLL import
name, import library behavior, and native modules must be compatible with that
compiler family.

Wine is required for the build-time linked-runtime probe and fast feedback, but
a Windows VM remains part of the release gate. Windows command-line quoting,
DLL lookup, filesystem semantics, and endpoint security behavior can differ
from Wine.

Generated Windows launchers and the onefile extractor currently use the narrow
Win32 process/path APIs. They preserve quoting for ACP-representable arguments,
but executable paths, temporary paths, or arguments containing characters that
cannot be represented by the active Windows ANSI code page are unsupported.
Use ASCII-compatible paths and arguments until the generated runtime moves to
the wide-character Win32 APIs.

Target-relative bundle paths also reject control bytes, drive/ADS syntax,
Windows reserved device names, trailing dots/spaces, and case-folded collisions.
macOS and Windows destination collisions are checked case-insensitively even
when the build host filesystem is case-sensitive. Non-ASCII target-relative
paths are rejected for these two targets because byte-wise Lua case conversion
cannot safely model their filesystem Unicode normalization and case folding.

[#platforms-native]
== Native Lua C modules

Discovery copies the module file selected from `package.cpath`. It does not:

* rebuild the module for another OS or architecture;
* change the Lua ABI exported by the module;
* rewrite its RPATH or install name;
* copy dependent OpenSSL, database, C++ runtime, or vendor libraries;
* verify the module's license or redistribution terms.

Use the source program's actual module paths during analysis:

[source,sh]
----
LUA_PATH='/srv/app/share/lua/5.4/?.lua;;' \
LUA_CPATH='/srv/app/lib/lua/5.4/?.so;;' \
luai -t app/main.lua --max-deps 250
----

Then inspect every copied native file:

[source,sh]
----
find build/app/.luai/native -type f -print
file build/app/.luai/native/*
ldd build/app/.luai/native/*.so
----

Do not assume that a successful launcher compile proves native closure. Native
load errors occur when the application first requires the module.

[#platforms-limits]
== Explicit limits

The current release does not provide:

* universal cross-platform executables;
* automatic native dependency closure;
* bytecode-only packaging;
* encryption or source obfuscation;
* loading native modules directly from onefile without extraction;
* Windows ARM or 32-bit Windows profiles;
* Linux-to-macOS compilation;
* macOS code signing or notarization;
* MSI, PKG, DEB, RPM, or other installer generation.

These are product boundaries. Treat them as unsupported rather than features
that may happen to work.
