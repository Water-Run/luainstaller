= Test programs and fixtures

The `test/` tree contains both contract tests and small applications used as
packaging targets. The applications are intentionally more realistic than unit
fixtures: they exercise multi-file discovery, native modules, persistent data,
HTTP startup, clean environments, and onefile extraction.

== Test entry points

`cli_split_smoke.lua`::
Checks the two CLI grammars and their error output.

`contract_docs.lua`::
Turns documented behavior into executable contracts. It covers discovery,
manual include/exclude precedence, option validation, output ownership,
manifests, bundled searcher order, logs, and documentation navigation.

`production_edges.lua`::
Checks production failure boundaries: syntax and lexer edge cases, checked I/O,
source snapshots, recursive SHA-256 ownership, target paths and Lua ABI,
deterministic onefile extraction, concurrent logs, LuaRocks installation, and
remote-script safety.

`smoke_all.lua`::
Runs the comprehensive local audit: syntax, samples, API, generated C,
installation layouts, onedir, onefile, native dependencies, safety regressions,
and clean-environment execution.

`version_contract.lua` and native integration tests::
Verify the official Lua 5.1 through 5.5 contract, the selected native compiler
and Lua ABI, onedir and onefile output, deterministic extraction, installed
LuaRocks commands, and execution on a target whose `PATH` contains no Lua.

`support/`::
Shared fixture, process, loader, and assertion helpers for the contract suites.

Run the local contract suites from the repository root with the Lua interpreter
being tested (shown as `$LUA` below):

[source,sh]
----
$LUA test/cli_split_smoke.lua
$LUA test/contract_docs.lua
$LUA test/production_edges.lua
$LUA test/smoke_all.lua
----

Run the pinned compatibility matrix with the native toolchain on the current
physical host:

[source,sh]
----
HOST_LABEL=my-linux-or-mac sh tools/test-lua-versions.sh
----

[source,powershell]
----
powershell.exe -NoLogo -NoProfile -NonInteractive `
  -ExecutionPolicy Bypass -File .\tools\test-lua-versions.ps1 `
  -HostLabel my-windows
----

Both matrix runners download the official Lua 5.1.5, 5.2.4, 5.3.6, 5.4.8,
and 5.5.0 sources, verify pinned SHA-256 digests, and write per-version evidence
logs below a guarded `luainstaller-*` temporary directory. These are tested
versions, not a platform or patch-version allowlist.

== Packaging targets

`runtime_bundle/`::
Minimal entry plus one Lua module. Use it first for launcher and onefile work.

`single_file/`::
Thirty standalone Lua examples used for broad entry-script smoke coverage.

`student_management_system/`::
Multi-file CLI with JSON storage and `cjson`.

`savinglua/`::
SQLite-backed key/value program using `cjson` and `lsqlite3`.

`ltokei/`::
Filesystem source counter using LuaFileSystem.

`firebird_web_sql/`::
Authenticated HTTP application using Pegasus, LuaSocket, JSON, and optional
database support.

Each target directory documents its own runtime dependencies. A test that needs
a native module must state the Lua ABI, operating system, and architecture it
expects.

== Working with a target

Run the source before packaging it:

[source,sh]
----
LUA=${LUA:-lua}
$LUA test/runtime_bundle/main.lua direct
$LUA test/student_management_system/main.lua --help
----

Then analyze and build:

[source,sh]
----
luai -t test/runtime_bundle/main.lua --max-deps 120
luai -b --dir test/runtime_bundle/main.lua \
  -o build/runtime-test --max-deps 120
env -i PATH=/usr/bin:/bin build/runtime-test/runtime-test clean
----

Remote coverage is driven by:

[source,sh]
----
sh tools/remote-test-linux.sh
MAC_HOST=yymac06 sh tools/remote-test-macos.sh
MAC_HOST=yymacstudio sh tools/remote-test-macos.sh
----

[source,powershell]
----
powershell.exe -NoLogo -NoProfile -NonInteractive `
  -ExecutionPolicy Bypass -File .\tools\test-lua-versions.ps1
----

Remote scripts accept destructive paths only below normalized
`/tmp/luainstaller-*` roots, transfer a `git archive` snapshot, and verify pinned
source archives. Linux and macOS orchestration targets physical lab machines.
Windows runs locally on the physical Windows host with PowerShell, MSVC, and the
Windows SDK; it does not use SSH, Wine, MinGW, a VM, or cross-compilation.

== Adding coverage

* Prefer the smallest fixture that reproduces the behavior.
* Add a regression assertion before changing production code.
* Put shared mechanics in `test/support/`.
* Keep sample dependencies explicit and reproducible.
* Test generated bundles with Lua environment variables cleared.
* For native or generated-C changes, run every affected remote architecture.
* Do not preserve obsolete compatibility samples without a supported release
  contract.
